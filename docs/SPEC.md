## 1. Product Intent

### 1.1 Problem

Psychiatric care often has long gaps between appointments. During that time the doctor lacks structured visibility into:

- how the patient feels between visits
- whether symptoms are improving or worsening
- whether prescribed medication is actually being taken
- whether side effects or safety issues appeared after the last visit

As a result, the next appointment relies too heavily on recall instead of observed dynamics.

### 1.2 Product Goal

`Docassist` should give the doctor a compact, reliable picture of the patient's condition between visits and give the patient a simple way to report what happened during that period.

The first production release should solve one clear job:

> A psychiatrist can review one patient's recent questionnaires, medication adherence, and reported side effects before or during the next appointment without reconstructing the history from memory alone.

### 1.3 Target Users

| User | Primary job |
|------|-------------|
| `doctor` | Monitor assigned patients between visits and review recent clinical data quickly |
| `patient` | Submit questionnaires, record medication adherence, and report side effects with minimal friction |
| `admin` | Operate the system, manage accounts, and support safe access control |

---

## 2. MVP Outcome

### 2.1 MVP Definition

The MVP is successful if it delivers the following end-to-end workflow:

1. A doctor creates or activates a patient account.
2. A doctor assigns one or more supported questionnaires and records current medication.
3. A patient logs in from the web app, completes assigned questionnaires, logs adherence, and reports side effects.
4. A doctor opens the patient record and sees recent submissions in one place.
5. The system highlights urgent safety items that should not wait until the next appointment.

### 2.2 MVP Success Criteria

| Outcome | Target |
|---------|--------|
| Doctor can understand a patient's recent status quickly | review in under 2 minutes |
| Patient can complete a scheduled questionnaire without support | standard web flow works on mobile and desktop |
| Medication adherence is recorded in a structured form | doctor can review recent adherence history |
| Urgent safety issues are surfaced reliably | critical items appear in doctor view without manual data digging |

### 2.3 What MVP Must Not Become

The MVP is not a full clinical decision support platform, not a replacement for appointments, and not an EMR.

It should avoid:

- diagnosis suggestions
- treatment recommendations generated by AI
- complex analytics that are hard to validate clinically
- broad multi-specialty expansion before the psychiatry workflow is stable

---

## 3. MVP Scope

### 3.1 In Scope For MVP

- secure login for `admin`, `doctor`, and `patient`
- doctor-managed patient roster
- patient accounts linked to exactly one doctor in MVP
- doctor-managed medication list for each patient
- patient medication adherence logging
- patient side effect reporting
- questionnaire assignment and submission
- initial questionnaire support for `PHQ-9` and `GAD-7`
- doctor patient view with recent questionnaire scores, adherence history, and side effects
- critical safety highlighting for clearly defined cases
- basic email-based account and reminder flows if needed for core onboarding

### 3.2 Out Of Scope For MVP

- AI-generated clinical advice
- EMR/EHR integrations
- e-prescribing or pharmacy integrations
- native mobile apps
- patient-to-patient or doctor-to-patient chat
- multi-clinic shared records
- advanced analytics beyond simple trends and clearly defined safety signals
- specialty-specific UIs beyond psychiatry

### 3.3 Deferred But Preserved

Features intentionally deferred from MVP are tracked in [SPEC_POST_MVP.md](/home/niquetamerewsl/projects/docassist/docs/SPEC_POST_MVP.md).

## 4. Core User Flows

### 4.1 Doctor Flow

1. Log in.
2. Open patient roster.
3. Create or activate a patient.
4. Record current medications.
5. Assign `PHQ-9` and/or `GAD-7`.
6. Review recent submissions, adherence, side effects, and safety flags before the next visit.

### 4.2 Patient Flow

1. Log in.
2. See pending tasks.
3. Complete assigned questionnaire.
4. Log whether medication was taken.
5. Report any side effects.

### 4.3 Admin Flow

1. Manage user access.
2. Deactivate accounts if needed.
3. Support safe system operation without accessing unnecessary clinical detail.

## 5. Domain Rules

These rules are part of the product contract for MVP.

### 5.1 Access Rules

- A doctor can access only patients assigned to that doctor.
- A patient can access only their own records.
- An admin manages accounts and operations, but should not require routine access to patient clinical content.

### 5.2 Questionnaire Rules

- MVP supports `PHQ-9` and `GAD-7`.
- Questionnaire scoring must follow the validated scoring model for each instrument.
- Raw answers and total score must be stored so the doctor can review history later.
- If a response contains a defined safety signal, the doctor view must clearly surface it.

### 5.3 Medication And Adherence Rules

- A doctor records the prescribed medication and dosage context for a patient.
- A patient can log whether the medication was taken and optionally record deviations.
- Adherence history must remain visible to the doctor as time-based records, not only as the latest status.

### 5.4 Side Effect Rules

- A patient can report a side effect with severity and free-text context.
- Severe side effects must be visible in the doctor workflow as priority items.

### 5.5 Safety Rules

The MVP should support a small set of explicit, auditable safety triggers rather than a large alert engine.

Required MVP safety signals:

- questionnaire responses that include suicidal ideation or another explicitly configured safety answer
- severe side effect reports above the configured threshold
- optionally overdue critical follow-up states if they can be implemented simply and tested reliably

## 6. Product Constraints

### 6.1 UX Constraints

- The web app must work on desktop and mobile browsers.
- Patient actions should be simple enough to complete in a few minutes.
- The doctor view should prioritize clarity over dense analytics.

### 6.2 Security And Privacy Constraints

- authentication and authorization are required from the first production release
- secrets must never be hardcoded
- patient data access boundaries must be enforced server-side
- logs must avoid leaking unnecessary clinical detail

### 6.3 Delivery Constraints

- The project should ship in thin vertical slices.
- Every phase must end in a usable state.
- The MVP should prefer fewer reliable features over a broad but weak platform.

---

## 7. Release Strategy

### 7.1 Recommended MVP Phases

| Phase | Outcome |
|------|---------|
| `PHASE_01` | Foundation, auth, user roles, project skeleton |
| `PHASE_02` | Doctor-patient relationship and patient account onboarding |
| `PHASE_03` | Medication tracking and adherence logging |
| `PHASE_04` | Questionnaire assignment, submission, and scoring for `PHQ-9` and `GAD-7` |
| `PHASE_05` | Doctor patient view with recent history and safety highlighting |
| `PHASE_06` | Production hardening, reminders, and MVP release readiness |

### 7.2 Scope Discipline

If a feature does not materially improve the core between-visit monitoring loop for the first psychiatry workflow, it should stay out of MVP and move to the deferred scope file.

## 8. Open Questions To Resolve During Planning

- Should admins be fully blind to clinical content, or do they need narrowly scoped support access?
- Is email enough for the first reminder flow, or should reminders wait until after MVP?
- Should the first release allow one patient to be linked to more than one doctor? The current recommendation is no.
- Which side effect severity threshold should count as a doctor-facing critical alert in production?

---

## 9. Summary

`Docassist` should start as a focused psychiatry monitoring product, not as a full clinical platform.

The MVP should deliver:

- patient-reported questionnaires
- medication adherence logging
- side effect reporting
- a doctor-facing patient summary
- a minimal set of reliable safety signals

Everything more ambitious should remain visible, but explicitly deferred.
